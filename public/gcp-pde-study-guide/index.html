CASE STUDY A

"Flowing Retail" — Omnichannel Sales

Transitioning a global retail transactional mainframe and POS network into an event-driven ledger with real-time inventory alerts and historical predictive warehousing.

💡 Architectural Pipeline

INGEST
Datastream & Pub/Sub

Datastream captures legacy database transactions (CDC); Pub/Sub ingests POS web sales streams.

PROCESS
Cloud Dataflow

Deduplicates order events, anonymizes/strips PII, and enriches fields on-the-fly.

STORE
Spanner & BigQuery

Spanner holds global ledger with relational consistency. BigQuery serves as data warehouse.

🛡️ Service Selection Criteria

  • Cloud Spanner: Selected for order ledger to ensure absolute global ACID transactional consistency and relational structure during massive peak retail spikes.
  • Cloud Datastream: Selected for seamless, serverless PostgreSQL-to-BigQuery Change Data Capture (CDC) replication without batch files or cron-job loaders.
  • BigQuery ML (BQML): Trains ARIMA_PLUS demand forecasting models directly inside the database using SQL, speeding up time-to-insight.
⚠️ Exam Focus: Spanner is chosen when global transaction consistency is required. If ACID is not needed but extreme analytical scale or simple key-value reads are required, choose BigQuery or Cloud SQL/Bigtable instead.
CASE STUDY B

"Earthquake Sensor Network" — IoT Streams

Ingesting, processing, and storing continuous, massive, high-frequency vibrational telemetry from 50,000 IoT seismic sensors for real-time alert trigger webhooks and historical scientific modeling.

💡 Architectural Pipeline

INGEST
Cloud Pub/Sub

Streams multi-million writes/sec seismological telemetry without custom scaling nodes.

PROCESS
Cloud Dataflow

Computes average tremors using 10s sliding windows and event-time watermark tracking.

STORE
Bigtable & BigQuery

Bigtable stores raw, high-throughput time-series metrics. BigQuery runs deep analytics.

🛡️ Service Selection Criteria

  • Cloud Bigtable: Chosen for high-frequency raw metric storage due to its sub-millisecond, massive linear write throughput scaling. Row key designed as station_id#timestamp_reversed to prevent tablet hotspotting.
  • Cloud Dataflow Windowing: Utilizes Sliding Windows (10s duration, sliding every 1s) to evaluate sudden seismic thresholds. Configured with withAllowedLateness to handle delayed data from remote, poorly connected base stations.
  • Cloud Run Trigger: Real-time warnings are triggered directly from Dataflow streams pushing alert messages to an auxiliary warning webhook topic.
⚠️ Exam Focus: Bigtable is optimal for timeseries writes, but row key design is critical. Never use timestamps as prefixes for keys if writes are sequential. Always place high-entropy keys (like station ID) or hashes first.
